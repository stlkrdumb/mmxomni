## Complexity Assessment

```
Complexity: Medium
Rationale: A single MCP server wrapping four bounded media endpoints (image / TTS / music / video) plus a small bonus set, with shared HTTP client, config, and packaging — about a dozen files, no DB or auth subsystem, but real architectural decisions around async video, asset return semantics, and multi-host distribution.
```

## Goal

Ship an npm-installable Model Context Protocol server (`mmxomni`) that exposes MiniMax's image, text-to-speech, music, and video generation endpoints as MCP tools, so any MCP-aware harness (Open WebUI, Claude Desktop, Cursor, OpenAI Agent SDK, etc.) can use them with a MiniMax Token Plan API key.

## Context

- The workspace is empty except for a Node-style `.gitignore`. The directory name `mmxomni` is the intended npm package name.
- The reference is the MiniMax public API: `https://platform.minimax.io/docs/api-reference` (the canonical mmx-cli skill in the environment confirms the surface: image-01, speech-2.8-hd, music-2.5, MiniMax-Hailuo-2.3, vision, search, quota).
- Authentication is a Bearer API key (`MINIMAX_API_KEY`), with a region selector (`global` vs `cn`) that changes the base URL.
- mmx-cli conventions to preserve (so users with existing credentials do not have to reconfigure):
  - Env vars: `MINIMAX_API_KEY`, `MINIMAX_REGION`
  - Persistent config at `~/.mmx/credentials.json` (and `~/.mmx/config.json`) — we re-use the same files.
  - Exit-code-like semantics: auth=3, quota=4, timeout=5, content-filter=10.
- Distribution target is `npx mmxomni` (stdio MCP transport) for any host that supports MCP. Open WebUI supports MCP servers as stdio subprocesses; Claude Desktop and Cursor use the same JSON config block.
- Out of immediate scope (but tracked): making MiniMax a *default chat model* inside Open WebUI. That is an Open WebUI "OpenAI-compatible provider" concern, separate from this server. The plan's `Out of Scope` section calls this out.

## Assumptions

- The implementer has Node.js ≥ 20 available; the package will declare `"engines": { "node": ">=20" }`.
- The implementer will use the official MCP TypeScript SDK (`@modelcontextprotocol/sdk`) and `zod` for tool schema validation.
- The MiniMax public API base URLs follow mmx-cli's defaults: `https://api.minimax.io/v1` for `global` and a CN equivalent for `region=cn` (resolved at runtime by `src/config.ts`).
- A MiniMax Token Plan API key (`sk-...`) is supplied by the end user via env var or `~/.mmx/credentials.json`; the server does not bundle, generate, or proxy keys.
- Generated media assets are returned to the model as MCP content blocks. Images are returned as native MCP `Image` content (base64) by default; audio (TTS/music) and video are returned as text with a hosted URL plus an explicit `download` tool, so a single generated MP3/MP4 does not blow up the model's context window. An `embed=true` flag on each tool opts into base64 for clients that need it.
- Video generation is treated as an async task: the tool returns a `task_id` immediately, and a companion `mmx_video_status` tool plus `mmx_video_download` tool fetch progress and the final asset. A `--wait` / `wait_timeout_seconds` flag enables synchronous polling for simpler hosts.

## Acceptance Criteria

- AC-1: The repository contains a `package.json` named `mmxomni` with a `bin` entry of `mmxomni` → `dist/index.js`, `engines.node >= 20`, scripts `build`, `start`, `test`, `lint`, and dependencies `@modelcontextprotocol/sdk`, `zod`, and `undici`.
- AC-2: `npm run build` produces a working `dist/`; `npx mmxomni` (or `node dist/index.js`) starts an MCP server over stdio that responds to `initialize` and `tools/list` without error.
- AC-3: `tools/list` advertises at least: `mmx_image_generate`, `mmx_speech_synthesize`, `mmx_music_generate`, `mmx_video_generate`, `mmx_video_status`, `mmx_video_download`. Each tool declares a `description` and a JSON-Schema `inputSchema` (zod-derived) with required params and sensible defaults (model defaults to the mmx-cli canonical model for that modality).
- AC-4: Auth resolves in this precedence and only this precedence: CLI flag `--api-key` → `MINIMAX_API_KEY` env var → `MINIMAX_API_KEY` field in `~/.mmx/credentials.json` → `api_key` in `~/.mmx/config.json`. Region resolves the same way via `--region` / `MINIMAX_REGION` / config, defaulting to `global`. Missing key produces exit code 3 and a human-readable stderr message.
- AC-5: `mmx_image_generate` calls the MiniMax image endpoint with the documented `image-01` defaults, returns the result as an MCP `Image` content block (base64) when `embed=true` and as text URL(s) otherwise, and surfaces API errors as MCP tool errors with codes 1/3/4/10 mapped from the HTTP response.
- AC-6: `mmx_speech_synthesize` calls the TTS endpoint, accepts `text`, `voice`, `model`, `format`, `speed`, `volume`, `pitch`, `sample_rate`, `bitrate`, `language_boost`, and `save_path`. It returns text containing the audio URL by default, and an embedded `audio` MCP resource (base64) when `embed=true`.
- AC-7: `mmx_music_generate` accepts `prompt` and/or `lyrics` (validated so at least one is present), plus the full mmx-cli control set (`vocals`, `genre`, `mood`, `instruments`, `tempo`, `bpm`, `key`, `structure`, `references`, `avoid`, `use_case`, `instrumental`, `aigc_watermark`, `format`, `sample_rate`, `bitrate`, `save_path`). `instrumental=true` is rejected at the schema level if `lyrics` is also set.
- AC-8: `mmx_video_generate` is async-first: it submits a video task and returns a JSON text block with `{ task_id, status, model }`. A `wait=true` flag polls `mmx_video_status` at 5-second intervals up to a `wait_timeout_seconds` (default 600) and returns the resolved task when status is `succeeded`/`failed` or on timeout (exit code 5). `mmx_video_status` accepts `task_id` and returns the raw task object. `mmx_video_download` accepts `task_id` and `save_path`, fetches the file when status is `succeeded`, and otherwise errors.
- AC-9: Optional/bonus tools are present behind a feature flag: `mmx_vision_describe`, `mmx_search_query`, `mmx_quota_show`. `mmx_quota_show` is especially useful for Token Plan users to inspect remaining quotas. They are registered when their feature flag (`--enable-bonus` or `MMXOMNI_BONUS=1`) is set; otherwise they are absent from `tools/list`.
- AC-10: The shared HTTP client (`src/client.ts`) uses `undici`, applies the `Authorization: Bearer <key>` header, swaps the base URL by region, retries idempotent requests up to 3 times with exponential backoff on 429/5xx, normalizes MiniMax error bodies to `{ code, message, httpStatus }`, and maps them to MCP error codes 1/3/4/5/10 matching the mmx-cli exit-code table.
- AC-11: All logs and progress output go to **stderr**; **stdout** is reserved for the MCP transport (per MCP stdio spec). A `--log-level` flag (`error|warn|info|debug`) controls verbosity; default is `warn`.
- AC-12: Unit tests with Vitest cover: config precedence, error-code mapping, request/response shape for each tool (against an `undici` `MockAgent`), and the video async/poll lifecycle. A smoke test that hits the real API is gated on `MINIMAX_API_KEY` being set and otherwise skips. `npm test` exits 0 with no live network calls by default.
- AC-13: `README.md` documents: install (`npm i -g mmxomni` or `npx mmxomni`), auth setup (env var and `~/.mmx/credentials.json`), and copy-pasteable MCP config blocks for **Open WebUI** (Tools → MCP Servers), **Claude Desktop** (`claude_desktop_config.json`), **Cursor** (`.cursor/mcp.json`), and a generic stdio entry. It also lists the four core tools, their parameters, and one example call for each.

## Implementation Notes

- **Tech stack**: TypeScript ≥ 5.4, Node ≥ 20, `@modelcontextprotocol/sdk` (latest stable), `zod` for input validation, `undici` for HTTP, `pino` for stderr-only logging, `vitest` + `undici` `MockAgent` for tests, `tsup` (or `tsc`) for the build, `eslint` + `prettier` for style.
- **MCP transport**: stdio only for v1. SSE/HTTP can be added later behind a `--transport http` flag without breaking stdio clients.
- **Tool name convention**: `mmx_<verb>_<noun>` (e.g. `mmx_image_generate`). The `mmx_` prefix prevents collisions with the host's native tools (Open WebUI, Claude Desktop, etc. all already expose built-in tools).
- **Async video strategy**: Default to async + status tool (cheaper for long generations, works well with model reasoning about progress). Sync `wait=true` is an opt-in for hosts that cannot reason about a multi-step tool loop.
- **Config reuse**: Read `~/.mmx/credentials.json` and `~/.mmx/config.json` directly when they exist, so users who already configured `mmx-cli` do not duplicate settings. Do **not** mutate those files; mmxomni writes only to `~/.mmxomni/state.json` (e.g. last task IDs).
- **Asset return semantics**: keep the default text-URL path; offer `embed=true` per tool for clients that want base64 inline (Open WebUI's chat rendering, for instance, can show an `<img>` from a data URL). For very large video files, the model should always use `mmx_video_download` and a `save_path` — never embed video base64.
- **Region resolution**: `global` → `https://api.minimax.io/v1`; `cn` → `https://api.minimaxi.cn/v1`. Allow override via `--base-url` for testing/staging.
- **Versioning**: Pin to a specific MCP protocol version in `initialize` and bump it deliberately; surface it in `serverInfo.version` matching `package.json` version.
- **License**: MIT (no header requested by the user; the harness coding rules forbid adding license headers to source files).

## Out of Scope

- Making MiniMax a *default chat model* inside Open WebUI. That is configured in Open WebUI's **Settings → Models** by adding an OpenAI-compatible provider pointing at MiniMax's `/v1/chat/completions`. mmxomni is a tools server, not a chat-completions proxy; this can be a follow-up `mmxomni-chat` companion if there is demand.
- SSE / HTTP MCP transport (stdio only for v1).
- Streaming tool results (NDJSON over stdio). v1 returns final results; streaming is a v2 follow-up.
- Image-to-image / image-edit / video-to-video endpoints (v1 covers the four endpoints the user named plus the small bonus set).
- A web UI, dashboard, or hosted service. mmxomni runs locally as a subprocess spawned by the host application.
- Vercel / Cloudflare Pages deployment. This is a local CLI tool, not a web app, so the "Deploy" tab suggestion in the planning template does not apply. (If a hosted SSE variant is built later, that is a different artifact.)

## Deployment Suggestions & Platform Detection

- mmxomni is a **local Node.js CLI / MCP server**, not a web application. It is installed and run on the user's machine (or a server they control) and spawned by their chat host.
- **No web-host deploy step is needed or appropriate.** Distribution is via the public npm registry: `npm i -g mmxomni` or `npx mmxomni`. Users add it to their host's MCP config (examples in the README).
- If, in a future iteration, a hosted SSE variant is added, that *would* be a candidate for Vercel or Cloudflare Workers / Pages; for v1 we do not produce any web artifacts.
- For Open WebUI specifically, the install path is: **Settings → External Tools → Add MCP Server** with `command: npx`, `args: ["-y", "mmxomni"]`, and `env: { MINIMAX_API_KEY: "sk-..." }`. The README will include this block verbatim.
